# Emotion City OS 実装メモ

## 画面分離

- `resident.html`: 習志野市のスマホ用。住民向け問い合わせ送信。場所、カテゴリ、内容、体感不安度を入力。
- `admin.html`: 習志野市のPC管理者用。問い合わせ一覧、習志野市マップ、AI要約、優先度、担当課、部署横断共有を表示。
- `index.html`: 入口選択のみ。住民画面と管理者画面は同じHTML内の切替ではなく、別ページとして完全に分離。

## AI構成のおすすめ

- クラウド標準: `gpt-5.4-mini`
  - 大量の問い合わせに対する要約、タグ付け、危険度判定に向く。
  - コストと速度を重視する自治体SaaSの標準モデルにする。
- 重要案件・政策分析: `gpt-5.5`
  - 複雑な判断理由、複数部署にまたがる分析、議会・予算説明資料の作成に使う。
- ローカル予備: `gpt-oss` 系
  - 個人情報を庁内に閉じたい場合やネット障害時の継続運用に使う。

## ローカルとクラウドの切り離し

### Local

- 保存対象: 住民の原文、写真、位置、氏名や連絡先などの個人情報。
- 処理対象: 緊急度判定、要約、担当課候補、初動対応。
- 目的: 災害時やクラウド障害時でも住民対応を止めない。

### Cloud

- 保存対象: 匿名化済みタグ、地域メッシュ、カテゴリ、対応結果、統計値。
- 処理対象: モデル改善、自治体横断の傾向分析、予算最適化。
- 目的: 個人情報を出さずにAI精度と政策判断を改善する。

## 本番化する場合の構成

1. フロント: Next.js または Remix
2. API: Cloudflare Workers / Vercel Functions / AWS Lambda
3. DB: PostgreSQL
4. 認証: 住民は匿名またはSMS、管理者は自治体SSO
5. AI: Responses APIでクラウド推論、庁内GPUまたは閉域サーバーでローカル推論
6. 同期: 個人情報除去後のイベントだけをクラウドへ送信

## インストールした方がいいもの

- Node.js LTS: 本番用Webアプリ化、ビルド、API開発に必要。
- Git: チーム開発と履歴管理に必要。
- VS Code: HTML/CSS/JS、将来のNext.js開発に便利。
- PostgreSQL: 問い合わせ、対応履歴、タグ、部署共有の保存先。
- OpenAI API: クラウド側の要約、タグ付け、危険度判定、政策分析に使用。
- Ollama または vLLM: ローカルAI推論の検証に使用。
- Docker Desktop: DBやローカルAIを同じ構成で動かすために便利。
- Vercel または Cloudflare Pages: サイトを誰でも使えるように公開するために便利。

## デモ版の制約

- 現在のデモはブラウザ内 `localStorage` に保存するため、サーバー共有はしない。
- AI判定は本番APIではなく、キーワードベースの疑似判定。
- 公開する場合は静的ホスティングにアップロード可能。
